# Card-Draw Method Selection — Design Spec

**Date:** 2026-06-25
**Status:** Design approved, pending implementation plan

## Overview

Replace the vertical tile-based method selection screen (`MethodSelect.tsx`) with a card-drawing mechanic. At the start of each draw phase, 3 cards are dealt face-down. Affinity effects trigger (Will adds, Fate removes, Shadow shrouds), then cards flip face-up for player selection. Post-selection effects like Fate forcing a different pick also animate.

## Locked Design Decisions

| Decision | Choice |
|----------|--------|
| **Card form factor** | Tarot proportion portrait cards (2:3 ratio, ~120×180px), sized relative to viewport height |
| **Card back** | Celestial Veil — unified back for all cards: deep indigo with gold sun-moon mandala center, white constellation patterns at 4 cardinal edges, gold Art Deco corner brackets, light starfield |
| **Card fronts** | Per-minigame illustrated designs (to be generated by custom design model): Tarot, Dice, I Ching, Astral, Rune — each with unique thematic artwork, symbol, title, and flavor text |
| **Shrouded cards** | Shadow-veiled front — completely opaque dark veil makes identification impossible. Faint eye motif and "VEILED" label are the only visible elements. On selection, veil lifts in a dramatic animation revealing the actual card front before transitioning to minigame |
| **Layout** | Horizontal scrollable row, no card rotation, cards centered with gap between them. Swap button (Will's spreadRedraws) as a separate element below the spread |
| **Animation engine** | Pure Framer Motion (Approach A) — already the project's pattern |
| **Animation pacing** | Sequential but fast — each effect auto-resolves without player input. A global `EventBanner` component shows a text string at the top of the screen, the effect plays in the card spread, then advances automatically (~1.0-1.5s per effect) |
| **Fate force animation** | Card ascends on click (always). If Fate triggers: screen freeze (deliberate pause, not lag) → hand-of-fate descends and slams the chosen card back down → overlay text "Fate has marked another path" → the actual selected card rises with golden glow → transition to minigame |

## Component Architecture

### New Components

```
MethodSelect.tsx              ← Rewritten: orchestrates the full draw phase
  ├─ CardSpread.tsx                NEW: horizontal scroll container
  │   ├─ MethodCard.tsx            NEW: individual card (face-down, face-up, shrouded, animating states)
  │   └─ FateForceOverlay.tsx      NEW: screen-freeze + hand-of-fate animation
  └─ WillSwapButton.tsx            NEW: swap button below spread (extracted from MethodSelect)

EventBanner.tsx               NEW: global overlay component (sibling to InteractionSequencer in GameTable)
  └─ Renders at top of viewport, auto-advancing text messages
```

### Modified Components

```
GameTable.tsx                 ← Add EventBanner above screen router
GameEngine.ts                 ← Add initiateDrawPhase(), update selectMethod() return
types.ts                      ← Add drawPhaseState to GameState
```

### Component Responsibilities

**`MethodSelect`** — Owns the draw-phase lifecycle state machine. On mount, calls `engine.initiateDrawPhase()` to get the pool + effect reports. Drives the sequencing: dealing → animating effects → flip → await selection → post-selection.

**`CardSpread`** — Renders a horizontal flex container with `overflow-x: auto` and styled scrollbar. Receives `pool`, `shrouded`, `phase`, and callbacks. Handles scroll behavior.

**`MethodCard`** — Renders a single card with states: `face-down` (Celestial Veil back), `face-up` (per-type illustrated front), `shrouded` (shadow-veiled front), `selected` (ascending), `rejected` (dimmed post-Fate), `fated` (golden glow). All state transitions animated via framer-motion.

**`FateForceOverlay`** — Full-screen overlay shown during Fate force. Freeze effect (CSS `backdrop-filter` or overlay div to "pause" the scene), hand-of-fate SVG animation descending, rejected card shakes/dims, actual card glows. Auto-dismisses after ~2s.

**`EventBanner`** — Fixed position at top-center of viewport. Receives `{ messages: { text: string; animation?: string; affinity?: AffinityId }[] }`. Each message fades in, holds briefly, fades out, then advances. The `animation` key tells `CardSpread` which effect to play (widen/thin/shroud). The `affinity` key drives thematic styling (Will=green, Fate=gold, Shadow=purple).

**`WillSwapButton`** — Rendered below the card spread when `state.affinityEffects.spreadRedraws >= 1`. Calls `engine.swapMethod()` which generates a fresh pool and restarts the draw phase.

## Animation Timeline

### Phase 1: Deal (entry)
1. Cards fly in from bottom of screen, landing face-down in the spread
2. Subtle stagger (50-80ms per card)
3. Cards settle with a gentle bounce ease

### Phase 2: Pre-selection Effects (auto-sequenced)
For each `EffectReport` returned by `initiateDrawPhase()`:
1. `EventBanner` shows the effect text (e.g., "Your will widens the path — another way opens" in Will-green)
2. `CardSpread` plays the matching animation:
   - **Widen (Will)**: A new card slot materializes at the edge of the spread, card flies in face-down with green-tinted glow
   - **Thin (Fate)**: A card dims with golden light and dissolves/fades out, remaining cards re-center
   - **Shroud (Shadow)**: Dark purple-black tendrils creep across a face-up card and solidify into the opaque veil
3. Wait 1.0-1.5s, then advance to next effect
4. Total time for effects: ~1-3 seconds (usually 0-2 effects fire)

### Phase 3: Flip
1. All non-shrouded cards flip simultaneously (3D Y-axis rotation, ~400ms)
2. Shrouded cards flip but land on the veiled front (same timing)
3. Cards settle, cursor changes to pointer

### Phase 4: Selection
1. Player hovers a card — card lifts slightly (scale + translateY)
2. Player clicks a card — card ASCENDS (rises ~20px with scale 1.05, golden edge glow)
3. **Normal path**: Card ascends fully, then the screen transitions to the minigame
4. **Fate override path**: See Phase 5

### Phase 5: Fate Force (post-selection override)
1. Card ascends on click (same as normal — this always happens)
2. Screen FREEZES: a dark semi-transparent overlay with `backdrop-filter: blur(2px)` snaps in (~100ms), giving a deliberate "time stops" feel
3. Overlay text fades in at center: "Fate has marked another path" in Fate-gold
4. Hand-of-fate (golden spectral hand SVG) descends from top of screen, SLAMS the ascending card back down into the spread
5. The chosen card shakes and dims (rejected state)
6. A DIFFERENT card in the spread rises with golden glow (fated state)
7. Total duration: ~2-2.5s
8. Overlay and banner dismiss, transition to the fated card's minigame

### Phase 6: Shrouded Reveal (if applicable)
1. If the selected (or fated) card was shrouded, the veil lifts BEFORE minigame transition
2. Shadow wisps tear away from center outward, revealing the actual card front beneath
3. Duration: ~600ms

## Engine Integration

### New Method: `GameEngine.initiateDrawPhase()`

Called by `MethodSelect` on mount. Runs the dispatch chain synchronously:

```
select:draw:start  →  (Will widen, Fate thin via STRUCTURAL responders)
                   →  generatePool(target)
select:draw:end    →  (Shadow shroud via MUTATE responder)
```

Returns:
```typescript
{
  pool: DivinationType[];
  shrouded: number[];
  effectReports: EffectReport[];  // ordered, for EventBanner
}
```

### Modified Method: `GameEngine.selectMethod(index)`

Current behavior: dispatches `select:pick`, sets `selectedMethod`, transitions to `minigame`.

New behavior:
```
select:pick → (Fate force may redirect index via OVERRIDE responder)
            → returns { actualIndex, method, wasForced, forceReport }
```

`MethodSelect` uses `wasForced` to trigger the Fate force animation before calling the minigame transition.

### State Addition

```typescript
// In GameState:
drawPhase: {
  pool: DivinationType[];
  shrouded: number[];
  effectReports: EffectReport[];
} | null;
```

`drawPhase` is set by `initiateDrawPhase()` and cleared when the player transitions to a minigame. It exists only during `screen === 'method-select'`.

### EffectReport Shape (existing type, reused)

```typescript
interface EffectReport {
  responderId: string;
  label: string;        // affinity name for EventBanner: "Will", "Fate", "Shadow"
  description: string;  // flavor text for EventBanner
  animation: string;    // 'widen' | 'thin' | 'shroud' | 'override'
  targetSlot?: number;  // which card index the effect targets
}
```

## EventBanner State

```typescript
// Managed by GameTable, pushed to by any screen:
interface BannerMessage {
  text: string;
  affinity?: 'will' | 'fate' | 'shadow' | 'light' | 'chaos' | 'order';
  animation?: string;  // passed to current screen's animation handler
  duration?: number;   // override default display time (ms)
}
```

`GameEngine` gets a new method: `pushBannerMessage(msg: BannerMessage)` which adds to `state.bannerQueue`. `EventBanner` consumes from the queue, auto-advancing.

## Edge Cases

1. **Pool size 1**: If Fate thins to a single card, still render it as a card (not auto-select). Player must click it to proceed.

2. **Pool size > 5**: Horizontal scroll activates. Scrollbar styled to match theme (thin, dark, subtle). Auto-scroll to reveal newly added cards.

3. **All cards shrouded**: Theoretically possible if Shadow is Dominant and RNG is favorable. All cards flip to veiled fronts. Player must choose blind. Shrouded reveal plays regardless of which is picked.

4. **Will swap mid-phase**: If the player clicks "Call for different methods", the entire draw phase resets — cards fly away, new cards deal in, effects re-trigger. The swap counts against `spreadRedraws`.

5. **Happening interruption**: Happenings interrupt BETWEEN minigames, not during method selection. So the card-draw phase never coexists with a happening.

6. **Mobile (≤480px)**: Cards scale down to fit 3 cards visible (~90×135px). Horizontal scroll still works. EventBanner text is smaller. Fate force overlay adapts to smaller viewport.

7. **Fast clicking**: Once a card is clicked (ascend begins), all cards become non-interactive (`pointer-events: none`) until the transition completes.

8. **No effects**: The common case — no affinity effects fire. Cards deal → flip → player picks. No EventBanner messages shown, no animations played. The phase transitions directly from deal to flip.

9. **Debug panel forcing**: When a debug scenario forces an effect, it bypasses the `roll` but not the `condition`. The effect always produces an `EffectReport`, so it always appears in the banner + animation sequence.
